<blockquote> <br> <p>AI创作声明：以下内容由大模型总结自我与大模型的<a href="https://github.com/LetMeFly666/LeetCode/pull/1519">对话</a>。</p> <br></blockquote> <br><h2><a id="CORS__Origin__null___Nginx__3"></a>CORS 跨域重定向后 Origin 变 null —— 一次 Nginx 字体加载失败的排查记录</h2> <br><h3><a id="_5"></a>现象</h3> <br><p>博客 <code>blog.letmefly.xyz</code> 页面加载 MathJax 数学公式字体，前端引用地址是 <code>https://letmefly.xyz/Links/JS/MathJax/.../MathJax_Zero.woff</code>。Nginx 会将 <code>letmefly.xyz</code> 的请求 302 重定向到 <code>web.letmefly.xyz</code>。浏览器控制台报出 CORS 错误：</p> <br><pre><code>Access to font at 'https://web.letmefly.xyz/...' (redirected from 'https://letmefly.xyz/...')<br>from origin 'https://blog.letmefly.xyz' has been blocked by CORS policy:<br>No 'Access-Control-Allow-Origin' header is present on the requested resource.<br></code></pre> <br><h3><a id="_15"></a>排查过程</h3> <br><h4><a id="_Nginx_CORS__17"></a>第一步：确认 Nginx CORS 配置</h4> <br><p>Nginx 使用全局 <code>map</code> 变量做 CORS 白名单：</p> <br><pre><code class="prism language-nginx">map $http_origin $corsHost {<br>    default "";<br>    "~https://blog.letmefly.xyz" https://blog.letmefly.xyz;<br>    # ...其他域名<br>}<br></code></pre> <br><p>各 server 块通过 <code>add_header Access-Control-Allow-Origin $corsHost;</code> 输出 CORS 头。看起来没问题。</p> <br><h4><a id="curl__31"></a>第二步：curl 验证最终目标</h4> <br><p>直接带 Origin 请求 <code>web.letmefly.xyz</code>：</p> <br><pre><code class="prism language-bash"><span class="token function">curl</span> <span class="token parameter variable">-I</span> <span class="token parameter variable">-H</span> <span class="token string">"Origin: https://blog.letmefly.xyz"</span> <span class="token punctuation">\</span><br>  https://web.letmefly.xyz/Links/JS/MathJax/output/chtml/fonts/woff-v2/MathJax_Zero.woff<br></code></pre> <br><p>返回了 <code>access-control-allow-origin: https://blog.letmefly.xyz</code>，完全正确。</p> <br><h4><a id="curl__42"></a>第三步：curl 验证重定向源</h4> <br><pre><code class="prism language-bash"><span class="token function">curl</span> <span class="token parameter variable">-I</span> <span class="token parameter variable">-H</span> <span class="token string">"Origin: https://blog.letmefly.xyz"</span> <span class="token punctuation">\</span><br>  https://letmefly.xyz/Links/JS/MathJax/output/chtml/fonts/woff-v2/MathJax_Zero.woff<br></code></pre> <br><p>302 响应也带了正确的 <code>access-control-allow-origin: https://blog.letmefly.xyz</code>。</p> <br><p>两端 curl 都正确，但浏览器就是报错。这说明<strong>浏览器实际发出的请求和 curl 模拟的不一样</strong>。</p> <br><h4><a id="Origin__null_53"></a>第四步：根因定位——Origin 变成了 null</h4> <br><p>综合以上线索：curl 手动带 Origin 请求没问题，但浏览器报错。说明浏览器跟随 302 重定向后，发到 <code>web.letmefly.xyz</code> 的请求里 <strong>Origin 不是 <code>https://blog.letmefly.xyz</code></strong>。那它变成了什么？答案是字符串 <code>"null"</code>。</p> <br><h3><a id="WHATWG_Fetch__redirecttainted_origin_57"></a>根因：WHATWG Fetch 规范的 redirect-tainted origin</h3> <br><p>这不是浏览器 bug，而是 <strong>WHATWG Fetch 规范</strong>明确要求的行为。</p> <br><p>规范定义了 <strong>redirect-tainted origin</strong>（重定向污染源，旧称 tainted origin flag）机制：</p> <br><ul><li><a href="https://fetch.spec.whatwg.org/#http-redirect-fetch" rel="nofollow">Fetch 规范 Section 4.5 HTTP-redirect fetch</a> 第 10 步：如果重定向目标 URL 的 origin 与请求当前 URL 的 origin <strong>不同</strong>，且请求的 origin 与当前 URL 的 origin 也<strong>不同</strong>，则设置请求的 tainted origin flag。</li><li><a href="https://fetch.spec.whatwg.org/#serializing-a-request-origin" rel="nofollow">Serializing a request origin</a> 算法：如果请求的 tainted origin flag 已设置，则返回字符串 <code>"null"</code>。</li></ul> <br><p>翻译成人话：<strong>当一个 CORS 请求经历了跨域重定向（重定向前后的 origin 不同），浏览器会把后续请求的 <code>Origin</code> 头设为字符串 <code>"null"</code>。</strong></p> <br><p>在这个案例中，请求链是：</p> <br><pre><code>blog.letmefly.xyz  →(fetch)→  letmefly.xyz  →(302)→  web.letmefly.xyz<br>     origin                    host A                    host B<br></code></pre> <br><p><code>letmefly.xyz</code> → <code>web.letmefly.xyz</code> 是跨域重定向，触发 tainted origin flag。浏览器向 <code>web.letmefly.xyz</code> 发送的请求中 <code>Origin: null</code>。Nginx 的 <code>map $http_origin</code> 匹配不到 <code>null</code>，走了 default <code>""</code>，不输出 CORS 头，浏览器拦截。</p> <br><p><strong>为什么规范要这样设计？</strong> 出于隐私和安全保护。举个例子：假设你在 <code>trusted-bank.com</code> 的页面上，页面向 <code>api.trusted-bank.com</code> 发了一个带 Cookie 的请求。如果 <code>api.trusted-bank.com</code> 被攻击者控制（或攻击者通过 DNS 劫持把请求引到了 <code>evil.com</code>），而浏览器仍然把 <code>Origin: https://trusted-bank.com</code> 原封不动地传过去，那么 <code>evil.com</code> 收到这个请求后就能冒充来自 <code>trusted-bank.com</code> 的合法请求——这就是<a href="https://en.wikipedia.org/wiki/Confused_deputy_problem" rel="nofollow">混淆代理攻击（confused deputy attack）</a>。浏览器把跨域重定向后的 Origin 设为 <code>null</code>，就是为了切断这条信任链：重定向目标不应该自动继承原始请求的可信身份。</p> <br><h3><a id="_79"></a>各浏览器实现情况</h3> <br><table><thead><tr><th>浏览器</th><th>行为</th><th>参考</th></tr></thead><tbody><tr><td>Chrome</td><td>跨域重定向后 <code>Origin: null</code></td><td><a href="https://issues.chromium.org/40290910" rel="nofollow">Chromium Issue 154967</a></td></tr><tr><td>Firefox</td><td>跨域重定向后 <code>Origin: null</code></td><td><a href="https://bugzilla.mozilla.org/show_bug.cgi?id=1444278" rel="nofollow">Bug 1444278</a>（Firefox 后续版本已对齐规范）</td></tr><tr><td>Safari</td><td>跨域重定向后 <code>Origin: null</code></td><td>同规范行为</td></tr></tbody></table><br><p>所有现代浏览器都遵循这一规范行为。Firefox 的 Bug 1444278 记录了完整的修复过程：之前 Firefox 在跨域重定向后仍然发送原始 Origin（与 Chrome/Safari 不一致），后来修复为发送 <code>"null"</code> 以对齐 WHATWG Fetch 规范。规范维护者 annevk 在该 bug 中确认：「Once we cross origin boundaries the request’s origin is supposed to become a unique opaque identifier (which serializes to null).」</p> <br><h3><a id="_89"></a>解决方案</h3> <br><p><strong>最佳方案（推荐）</strong>：在前端直接引用最终地址 <code>https://web.letmefly.xyz/...</code>，避免经过 <code>letmefly.xyz</code> 的重定向，根本不触发 tainted origin flag。</p> <br><p><strong>备选方案一</strong>：在 <code>web.letmefly.xyz</code> 的 Nginx 配置中，对字体文件用 <code>*</code> 通配符：</p> <br><pre><code class="prism language-nginx">location ~* \.(woff|woff2|ttf|eot|otf)$ {<br>    add_header Access-Control-Allow-Origin "*" always;<br>}<br></code></pre> <br><p>字体文件本身不涉及敏感数据，用 <code>*</code> 通配是安全的。</p> <br><p><strong>备选方案二</strong>：在 map 中加一条匹配 <code>null</code> 的规则：</p> <br><pre><code class="prism language-nginx">map $http_origin $corsHost {<br>    default "";<br>    "null" "https://blog.letmefly.xyz";<br>    "~https://blog.letmefly.xyz" https://blog.letmefly.xyz;<br>    # ...<br>}<br></code></pre> <br><p>但这样做安全性稍差——任何 Origin 为 null 的请求都会被放行为 <code>blog.letmefly.xyz</code>。</p> <br><h3><a id="Nginx_add_header__116"></a>排查过程中踩的另一个坑：Nginx add_header 继承陷阱</h3> <br><p>排查过程中还踩了一个坑：在 <code>web.letmefly.xyz</code> 的 server 块级别加了 <code>add_header Access-Control-Allow-Origin $corsHost;</code>，但不生效。</p> <br><p>原因涉及 Nginx <code>add_header</code> 的继承机制。根据<a href="https://nginx.org/en/docs/http/ngx_http_headers_module.html#add_header" rel="nofollow">官方文档</a>的描述：</p> <br><blockquote> <br> <p>These directives are inherited from the previous configuration level <strong>if and only if</strong> there are no <code>add_header</code> directives defined on the current level.</p> <br></blockquote> <br><p>也就是说，<code>add_header</code> <strong>默认是会继承上层的</strong>，但一旦当前层级（如 <code>location</code>）里出现了<strong>任何一条</strong> <code>add_header</code>，上层（<code>server</code>/<code>http</code>）的 <code>add_header</code> 就<strong>全部失效</strong>——不是合并，而是完全替换。</p> <br><p>以 <code>web.letmefly.xyz</code> 的配置为例，假设原本有这样的结构：</p> <br><pre><code class="prism language-nginx">server {<br>    server_name web.letmefly.xyz;<br>    add_header Access-Control-Allow-Origin $corsHost always;  # ← server 级别<br><br>    location / {<br>        # 这个 location 里没有任何 add_header → 会继承 server 级别的 CORS 头 ✓<br>        try_files $uri $uri/ =404;<br>    }<br><br>    location /api {<br>        add_header X-Frame-Options SAMEORIGIN;  # ← 一旦出现这条<br>        # server 级别的 add_header 全部失效，CORS 头丢失 ✗<br>    }<br>}<br></code></pre> <br><p>解决办法是把 CORS 头直接加到每个含有 <code>add_header</code> 的 location 块里。可以用 <code>include</code> 抽成公共片段避免重复：</p> <br><pre><code class="prism language-nginx"># /etc/nginx/snippets/cors.conf<br>add_header Access-Control-Allow-Origin $corsHost always;<br></code></pre> <br><p>然后在每个需要 CORS 头的 location 中引入：</p> <br><pre><code class="prism language-nginx">server {<br>    server_name web.letmefly.xyz;<br>    listen 443 ssl;<br>    root /srv/web/website;<br><br>    location / {<br>        include /etc/nginx/snippets/cors.conf;<br>        try_files $uri $uri/ =404;<br>    }<br><br>    location /api {<br>        include /etc/nginx/snippets/cors.conf;<br>        add_header X-Frame-Options SAMEORIGIN;<br>        proxy_pass http://backend;<br>    }<br><br>    location ~* \.(woff|woff2|ttf|eot|otf)$ {<br>        include /etc/nginx/snippets/cors.conf;<br>        expires 30d;<br>    }<br>}<br></code></pre> <br><p>这样即使某个 location 有自己的 <code>add_header</code>，CORS 头也不会丢失。</p> <br><blockquote> <br> <p><strong>补充</strong>：Nginx 1.29.3 新增了 <code>add_header_inherit merge;</code> 指令（<a href="https://nginx.org/" rel="nofollow">1.30.0 stable</a> 已包含），可以让子级别在保留自己 <code>add_header</code> 的同时继承上层的 <code>add_header</code>。</p> <br></blockquote> <br><hr />
