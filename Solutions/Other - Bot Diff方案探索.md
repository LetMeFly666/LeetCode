# 【LetMeFly】Bot Diff方案探索：如何让LLM稳定地修改文件

## 一、问题背景

在使用LLM（大语言模型）Bot自动化修改代码或文档时，我们面临一个核心难题：

- **从零创建**（a → b）：Bot表现良好，能准确生成目标内容
- **增量修改**（b → c）：Bot经常"乱改"——改错位置、遗漏修改、引入不必要的变更

这个问题在实际开发中非常常见，尤其是当文件较长、需要精确修改特定行时。

## 二、问题分析

### 2.1 为什么LLM在增量修改时容易出错？

1. **上下文窗口限制**：长文件超出有效注意力范围，LLM对文件中部和尾部内容的"记忆"变弱
2. **位置感知能力弱**：LLM擅长理解语义，但对"第37行"、"第2个代码块"等位置信息缺乏精确感知
3. **全量输出倾向**：LLM倾向于重新生成整个文件，而非精确表达"只改这一处"
4. **格式漂移**：在重新生成过程中，缩进、空行、注释格式等容易发生微妙变化

### 2.2 现有方案的局限

| 方案 | 优点 | 缺点 |
|------|------|------|
| 全量替换 | 简单直接 | 大文件易截断、易漂移 |
| 搜索替换 | 精确 | 多处相同文本时歧义 |
| 行号定位 | 直观 | LLM经常数错行号 |
| Unified Diff | 标准化 | LLM生成的diff经常格式错误 |

## 三、实验方案设计

本文将通过**实际的文件修改实验**来验证不同的diff策略，记录每种策略的成功率和失败模式。

### 3.1 实验方法

1. 准备一个包含多个section的测试文件
2. 对每个section提出明确的修改需求
3. 使用不同的diff策略执行修改
4. 验证修改结果是否符合预期
5. 记录成功/失败及原因

### 3.2 策略清单

- **策略A**：Unified Diff（标准 `--- a/file` / `+++ b/file` / `@@ -n,m +n,m @@` 格式）
- **策略B**：锚点定位法（用唯一上下文行作为锚点，指定在锚点前后插入/替换）
- **策略C**：分块Hash法（对文件分块计算hash，只传输变更块）
- **策略D**：语义区域法（用函数名/标题等语义标识定位修改区域）
