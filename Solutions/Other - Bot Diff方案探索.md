# 【LetMeFly】Bot Diff方案探索 - 如何让LLM精确修改文件

## 问题背景

在使用LLM Bot进行代码审查和自动修改时，我们发现一个普遍性问题：

- **第一次修改（a→b）**：Bot表现良好，能准确创建或修改文件
- **后续修改（b→c）**：Bot容易"乱改"，修改位置不准确，甚至破坏原有内容

这个问题在多个模型上都有体现，即使是Claude等"地表最强"模型也不例外。

## 问题分析

### 1. 为什么第一次修改容易成功？

第一次修改时，Bot可以使用"全量文件内容"模式（Full File Content），直接提供完整的文件内容。这种方式简单直接，不容易出错。

### 2. 为什么后续修改容易失败？

后续修改时，文件已经存在且内容可能较长。此时有两个困难：

- **上下文窗口限制**：大文件内容经过LLM的上下文窗口后容易被截断、总结或产生Unicode错误
- **Diff定位不准确**：LLM生成的unified diff中，上下文行可能与实际文件内容不完全匹配
- **多处修改干扰**：需要同时修改多个位置时，hunk之间的行号计算容易出错

### 3. 常见的"乱改"表现

| 问题类型 | 具体表现 | 根因 |
|---------|---------|------|
| 位置偏移 | 修改了错误的行 | 行号计算错误或上下文非唯一 |
| 内容截断 | 文件末尾内容丢失 | 上下文窗口溢出 |
| 重复内容 | 同一段内容出现两次 | Hunk边界重叠 |
| 无关修改 | 修改了不应该改的地方 | 上下文匹配模糊 |
| 静默失败 | API返回成功但实际未修改 | diff代码块干扰patch解析 |

## 方案探索

### 方案一：精确定位法

**核心思想**：每次diff只修改一个位置，使用足够多的上下文行来确保精确定位。

**优点**：定位准确，不容易出错
**缺点**：需要多次commit才能完成多处修改

### 方案二：分段Diff法

**核心思想**：将多处修改拆分为多个独立的hunk，每个hunk有充足的上下文。

关键要点：
1. 每个hunk至少包含3行上下文（前后各3行）
2. 相邻hunk之间至少间隔1行未修改内容
3. 行号必须精确计算，考虑前面hunk对行号的影响

### 方案三：锚点定位法

**核心思想**：利用文件中唯一性强的文本作为"锚点"来定位修改位置。

比如：
- 使用标题行（`## 某个标题`）作为锚点
- 使用函数签名作为锚点
- 使用特殊注释作为锚点

## 结论与最佳实践

经过多轮实验，总结出以下经验：

### ✅ 必须遵守的规则

1. **行号必须精确**：hunk header 中的行号必须与文件实际内容完全匹配。误差哪怕只有几行，都可能导致修改应用到错误位置。

2. **上下文行必须唯一**：diff 的上下文行应该在整个文件中是唯一的。如果多处出现相同文本，必须包含能区分位置的标题行或其他唯一标识。

3. **避免文件中包含 diff 示例**：当文件内容包含 `--- a/` 或 `+++ b/` 等 diff 格式文本时，会干扰 patch 解析器，导致 diff 模式完全失效。

4. **验证每次修改**：不能信任 API 返回的成功状态。必须在每次 diff 后获取文件内容并验证修改是否正确应用。

### ✅ 推荐的工作流

1. **修改前**：获取文件最新内容，精确计算行号
2. **构造 diff**：确保每个 hunk 的上下文包含唯一标识行
3. **提交 diff**：发送到 API
4. **验证结果**：通过 API 获取文件内容确认修改正确
5. **失败回退**：如果验证失败，使用 files 模式重写整个文件

### ✅ diff vs files 模式选择指南

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 新建文件 | files | 简单直接 |
| 小文件任意修改 | files | 不会出错 |
| 大文件小范围修改 | diff | 避免传输大量内容 |
| 文件含 diff 格式内容 | files | diff模式会被干扰 |
| 多 hunk 修改（Ⅴ2个） | diff | 可行，需精确行号 |
| 多 hunk 修改（≥3个且含插入） | 拆分为多次diff或用files | 行号偏移累积容易出错 |
| 修改失败后修复 | files | 直接覆盖最安全 |

### ✅ diff 质量检查清单

在发送 diff 前，检查以下几点：

- [ ] hunk header 行号是否精确？
- [ ] 上下文行是否在文件中唯一？
- [ ] 是否包含足够的上下文（建议至少3行）？
- [ ] 文件是否包含 diff 格式的代码块？如果是，改用 files 模式
- [ ] 多个 hunk 时，后续 hunk 的行号是否考虑了前面 hunk 的插入/删除？
- [ ] 文件末尾的修改是否特别处理（末尾 hunk 容易出错）？
