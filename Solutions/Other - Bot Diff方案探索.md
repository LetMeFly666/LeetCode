# 【LetMeFly】Bot Diff方案探索：如何让LLM稳定地修改文件

## 一、问题背景

在使用LLM（大语言模型）Bot自动化修改代码或文档时，我们面临一个核心难题：

- **从零创建**（a → b）：Bot表现良好，能准确生成目标内容
- **增量修改**（b → c）：Bot经常"乱改"——改错位置、遗漏修改、引入不必要的变更

这个问题在实际开发中非常常见，尤其是当文件较长、需要精确修改特定行时。如 [#1519](https://github.com/LetMeFly666/LeetCode/issues/1519) 和 [#1520](https://github.com/LetMeFly666/LeetCode/issues/1520) 中的经历所示，即使是当前最强的模型，在SKILL中限定"不要乱改"也无效。

## 二、问题深度分析

### 2.1 为什么LLM在增量修改时容易出错？

1. **上下文窗口限制**：长文件超出有效注意力范围，LLM对文件中部和尾部内容的"记忆"变弱
2. **位置感知能力弱**：LLM擅长理解语义，但对"第37行"、"第2个代码块"等位置信息缺乏精确感知
3. **全量输出倾向**：LLM倾向于重新生成整个文件，而非精确表达"只改这一处"
4. **行号计算能力差**：LLM在手写unified diff时经常计算错hunk header中的行号，尤其是多个hunk的情况

### 2.2 实测发现的关键因素

通过7轮实验，我们发现影响diff应用成功率的因素优先级为：

1. **Hunk数量**：单hunk不涉及行号累计偏移问题，最稳定
2. **行号精确度**：服务端工具对行号偏移有5行左右的容忍度，超过20行就会失败
3. **上下文行内容**：服务端支持模糊匹配，微小差异（中英文标点、额外空格）可容忍
4. **上下文行数量**：更多上下文行提高匹配准确度

## 四、核心结论与最佳实践

### 4.1 核心结论

> **LLM Bot做diff修改失败的根本原因是行号计算错误，而非工具链的问题。**
>
> 服务端的diff应用工具具有很强的容错能力（模糊匹配、行号偏移容忍、上下文文本微小差异容忍）。
> 只要保证行号偏移在合理范围内（≤ 5行），即使多hunk修改也能100%成功。

### 4.2 推荐的Bot Diff工作流

#### 方案A：单Hunk逐一提交（最稳定，推荐）

```
循环每个修改点：
  1. 通过API获取文件当前内容
  2. 确认目标行的精确位置和内容
  3. 生成单hunk diff，上下文行从实际文件提取
  4. 提交commit
  5. 验证修改结果
```

**优点**：
- 每次行号都基于最新文件，不存在累计偏移
- 单个修改失败不影响其他修改
- 每次修改都可以立即验证

**缺点**：
- commit数量较多，可能需要squash
- 每次都要获取文件，速度较慢

#### 方案B：本地difflib生成精确diff（高效，推荐）

```
1. 通过API获取文件当前内容
2. 在本地生成修改后的完整版本
3. 使用difflib计算精确unified diff
4. 一次提交多hunk diff
5. 验证修改结果
```

**优点**：
- 行号100%准确，无偏移风险
- 单次commit完成所有修改，干净整洁

**缺点**：
- 需要本地执行环境（sandbox）支持
- LLM生成修改版本时仍可能引入非预期变更

### 4.3 LLM写diff的要点清单

✅ **必须做**：
- 每次修改前获取文件最新内容
- 上下文行从实际文件中复制，不要凭记忆写
- 修改后立即验证结果
- 单hunk修改时提供至少3行上下文

⚠️ **应该做**：
- 使用程序化工具（difflib）生成diff，而非手写
- 多hunk时提供更多上下文行（6行+）
- 复杂修改拆分为多个小步骤

❌ **绝对不要**：
- 凭记忆手写多hunk diff的行号
- 在不知道当前文件内容的情况下写diff
- 一次提交多个文件的多hunk修改（复杂度爆炸）

## 五、实验详细记录

### 实验1：Unified Diff基准测试

- **策略**：3个hunk单次commit，行号手写
- **结果**：1/3成功
- **失败原因**：hunk header行号偏移5-11行，超出服务端容忍度，patch应用到错误位置

### 实验2：单Hunk逐一提交

- **策略**：拆分为3个独立的单hunk commit，每次刷新行号
- **结果**：3/3成功
- **关键发现**：每次基于最新文件行号，完全消除偏移问题

### 实验3：复杂场景单Hunk测试

- **策略**：5个独立的单hunk commit，覆盖代码块插入、多行删除、嵌套列表修改、引用块修改、代码内字符串修改
- **结果**：5/5成功
- **关键发现**：所有内容类型的修改均稳定可靠

### 实验4：difflib精确多Hunk测试

- **策略**：使用Python difflib生成精确diff，3个hunk单次commit
- **结果**：3/3成功
- **关键发现**：精确行号的多hunk diff完全可靠

### 实验5：行号偏移容忍度测试

- **策略**：故意在hunk header中写错行号（偏移3-5行），但提供6行上下文
- **结果**：2/2成功
- **关键发现**：服务端具有模糊匹配能力，小偏移可容忍

### 实验6：上下文微小差异容忍度测试

- **策略**：上下文行故意引入中英文标点差异、额外空格
- **结果**：2/2成功
- **关键发现**：服务端对上下文内容也有容忍度

### 实验7：极端偏移测试

- **策略**：行号故意偏移22行
- **结果**：0/1成功
- **失败现象**：patch在错误位置以插入方式应用，而非替换目标行

## 六、总结

Bot diff方案的核心不在于工具链，而在于**如何绕过LLM的位置感知短板**。

最实用的两个策略：

1. **单hunk逐一commit**：简单粗暴但100%可靠，适合修改点少的场景
2. **difflib自动计算**：高效且精确，适合多处同时修改的场景

两者可以结合使用：简单修改用方案A，复杂修改用方案B。
